# ATS Protocol Specification

## Overview

The Agentic Telemetry Standard (ATS) is a protocol that defines how AI coding agents interact with application telemetry. It consists of three components:

1. **Flow Graph** (`flow_graph.json`) — A JSON file that maps business logic flows to code (classes and methods).
2. **Runtime SDK** (`ATS.trace()`) — A lightweight tracing function embedded in application code.
3. **Agent Skill** (`SKILL.md`) — An instruction file that teaches AI agents how to use the protocol.

## Design Principles

- **AI-first**: The protocol is designed for AI agents, not human developers. Humans benefit indirectly.
- **Language-agnostic**: The `flow_graph.json` schema is universal. SDKs implement it per language.
- **Zero-overhead**: When a flow is inactive, trace calls must be no-ops with negligible cost.
- **Persistent knowledge**: The flow graph is committed to version control and grows over time.
- **IDE-agnostic**: No IDE-specific configuration (launch.json, tasks.json) should be required.

## Flow Graph Schema

The flow graph is a JSON file located at a configurable path (default: `.ats/flow_graph.json`).

### Required Fields

| Field | Type | Description |
|---|---|---|
| `ats_version` | string | Protocol version (semver) |
| `project` | string | Project name |
| `flows` | object | Map of flow name → flow definition |

### Flow Definition

| Field | Type | Required | Description |
|---|---|---|---|
| `description` | string | yes | What this flow represents |
| `active` | boolean | yes | Whether trace logs are enabled |
| `classes` | object | yes | Map of class name → list of method names |
| `tags` | string[] | no | Categorization tags |
| `known_issues` | string[] | no | AI-maintained list of known bugs |
| `last_debugged` | string | no | ISO 8601 date of last debug session |

### Naming Conventions

- Flow names: `SCREAMING_SNAKE_CASE` (e.g. `PAYMENT_FLOW`, `AUTH_FLOW`)
- Class names: Match the source code exactly (e.g. `PaymentService`)
- Method names: Match the source code exactly (e.g. `processPayment`)

### Example

```json
{
  "ats_version": "3.0.0",
  "project": "my_app",
  "flows": {
    "PAYMENT_FLOW": {
      "description": "Handles checkout and payment processing",
      "active": false,
      "classes": {
        "PaymentService": ["processPayment", "refund"],
        "CheckoutBloc": ["onCheckoutStarted", "onPaymentConfirmed"]
      }
    }
  }
}
```

## Runtime SDK Contract

Any ATS SDK implementation must provide:

### `trace(className, methodName, {data})`

- If the method belongs to an **active** flow → print a structured log line.
- If the method does NOT belong to any active flow → **no-op** (return immediately).
- Log format: `[ATS][FLOW_NAME] ClassName.methodName | {data}`

### `internalInit(methodMap, activeFlows)`

- Accepts a pre-compiled map of `"Class.method" → [flowNames]` and a list of active flow names.
- Used by the generated code to inject static mappings at startup.

## Configuration (`ats.yaml`)

An optional YAML file at the project root (similar to Flutter's `l10n.yaml`) that controls paths:

```yaml
ats-dir: .ats                           # Where flow_graph.json lives
output-dir: lib/generated/ats           # Where generated code is placed
output-ats-file: ats_generated.g.dart   # Generated file name
```

If this file is absent, defaults are used.

## CodeGen Architecture (V3)

Instead of reading JSON at runtime, ATS V3 compiles `flow_graph.json` into a native Dart file containing static `const` Maps. This means:

- No JSON parsing at runtime.
- Flow lookups are O(1) Map access.
- Changes take effect via Hot Restart (no full rebuild needed).

### Generated File Structure

```dart
// AUTO-GENERATED BY ATS CLI — DO NOT EDIT

import 'package:ats_flutter/ats_flutter.dart';

const _kMethodMap = <String, List<String>>{
  'PaymentService.processPayment': ['PAYMENT_FLOW'],
  'PaymentService.refund': ['PAYMENT_FLOW'],
};

const _kActiveFlows = <String>['PAYMENT_FLOW'];

abstract class AtsGenerated {
  static void init() {
    ATS.internalInit(_kMethodMap, _kActiveFlows);
  }
}
```

## Agent Skill Contract

An ATS skill file (`SKILL.md`) must instruct the AI agent to:

1. **Read** `flow_graph.json` at the start of every task.
2. **Map** new classes to the appropriate flow when writing or modifying code.
3. **Activate** flows when debugging (via `ats activate <FLOW>` or direct JSON edit + `ats sync`).
4. **Silence** flows when done (via `ats silence <FLOW>`).
5. **Instrument** classes with `ATS.trace()` when first encountered.
6. **Never remove** `ATS.trace()` calls — they are permanent instrumentation.
7. **Update** `updated_at` timestamp when modifying the graph.

## CLI Commands

| Command | Description |
|---|---|
| `ats init` | Set up ATS in a Flutter project |
| `ats sync` | Compile `flow_graph.json` into generated Dart code |
| `ats activate <FLOW>` | Set a flow active + auto-sync |
| `ats silence <FLOW>` | Set a flow silent + auto-sync |
| `ats status` | Show all flows and their active state |
| `ats flows` | List flows with their classes and methods |
| `ats skill install` | Install SKILL.md for AI agents |

## Versioning

- Protocol version follows semver.
- SDK versions are independent but must declare which protocol version they support.
- The `ats_version` field in `flow_graph.json` indicates which protocol version generated it.
